Deployment Strategy & CI/CD

We containerized the React frontend and Node backend as separate Docker images. Separating frontend and backend images lets us deploy backend changes independently, for example when adding API endpoints or performance improvements, without rebuilding the UI. The backend ran in Amazon ECS, AWS’s container orchestration service, inside a VPC, a network-isolated environment. ECS manages running containers and scales them as needed. An Application Load Balancer distributes incoming requests across containers, and the database lives in private subnets for security. I worked within this setup to ensure the APIs connected correctly and securely without managing the VPC itself. The frontend React app was built into static files, which include all the HTML, CSS, JavaScript, and assets the browser needs. These files were deployed to S3, AWS’s object storage service, and served through CloudFront, a content delivery network. This delivers content quickly to users worldwide and reduces load on the backend since static assets do not require API processing. Our CI/CD pipeline used GitHub Actions to run automated tests on pull requests. On merge, it built a versioned Docker image for the backend and pushed it to ECR, AWS’s container registry. ECS was then updated to use that exact image, ensuring staging and production ran the same artifact. Because each image was versioned, we could roll back to a previous tag quickly if needed.

  • Key Takeaway: Server is authoritative; clients only send move intents.

  • Client Responsibilities: Render board, send move intents (from-square → to-square)
  • Server Responsibilities:
    • Validate moves (legal & correct turn)
    • Update & persist game state
    • Broadcast results to clients via WebSockets
  • Communication / Sync: WebSockets push updated state to all clients in real-time
  • Security / Rules Enforcement: Auth identifies player; server enforces all game rules
  • Scaling / Performance Notes: Minimal, mostly focused on real-time sync

Designing a Real-Time Auction Platform

  • Key Takeaway: Backend is authoritative; clients submit bids and receive updates in real-time.
  • Core Models / Data: User, Auction, Item, Bid, BidHistory, AuctionStatus
  • Client Responsibilities: Display current auctions, submit bid intents, render real-time bid updates
  • Server Responsibilities:
    • Validate incoming bids (amount, timing, user eligibility)
    • Update and persist auction state, including highest bid and bidder
    • Broadcast auction updates to subscribed clients via WebSockets or server-sent events
    • Handle concurrent bids safely to prevent race conditions
    • Support creation and management of auctions, items, and bidding rules
  • Communication / Sync: WebSockets for live bid updates; REST APIs for auction creation, user management, and historical queries
  • Security / Rules Enforcement: Auth identifies bidders; server enforces bid validity, auction timing, and anti-cheat measures
  • Scaling / Performance Notes:
    • Use in-memory caching (Redis) for current auction state to reduce DB hits
    • Queue concurrent bid events to ensure sequential processing
    • Horizontally scale WebSocket servers with pub/sub for broadcasting updates

Balancing Technical Debt and Feature Delivery

Managing technical debt while delivering new features requires a strategic, balanced approach:

  • Prioritize debt: Identify critical debt that impacts performance, maintainability, or team velocity, and address it first.
  • Integrate into sprints: Allocate a portion of each sprint or release cycle specifically for refactoring and debt reduction.

By treating technical debt management as a continuous process rather than a one-time task, teams maintain velocity while keeping the codebase healthy and maintainable.

Redux

React & Redux: Large-Scale App Structure

  • Organize React apps by feature, grouping components, hooks, and types for better maintainability and scalability
  • useState for local component state and useEffect for side effects (API calls, subscriptions)
  • Use Redux for global state like authentication. It provides a predictable, centralized state management solution for complex React apps, making state changes explicit, simplifying debugging, and enabling consistent data flow across components, especially in large or highly interactive applications.
  • Split Redux store into feature slices; keep reducers pure (no side effects, no state mutation) and the root reducer simple.
  • Handle async logic using createAsyncThunk from Redux Toolkit.
  • Avoid redundant or computed data in the store; calculate derived values as needed.

Redux Async Flow (Client → Server → Client)

  1. User acts → dispatch an action (e.g., move piece, post, bid)
  2. Redux middleware (Thunk / RTK) handles async → calls API
  3. Server responds → returns data or error
  4. Redux reducer updates store with response
  5. Components auto re-render via useSelector
  6. Optional useEffect → side effects on state change (notifications, logging, etc.)

API & Networking

API Design, Scalability & Performance

  • RESTful endpoints: GET /users, POST /users, PUT /users/:id
  • Share TypeScript interfaces between backend and frontend: ensures consistent, type-safe API shapes at compile time. Example: interface User { id: number; name: string; email: string }. This prevents type errors in your code but does not stop bad data from reaching the backend.
  • Request validation with DTOs: enforces correct data at runtime on the backend. Example: @IsEmail() email: string; @IsString() name: string. Even if a client bypasses the frontend, DTOs prevent invalid or malicious input from being processed.
  • Secure endpoints (JWT + roles): Only admins can DELETE /users/:id
  • Handle large datasets: GET /posts?page=2&limit=20&authorId=5
  • Optimize queries: SQL join posts with comments; index user_id
  • Feature modules: users/ folder with controller, service, entity
  • Validation & error handling: Return 400 for missing fields, 500 for DB errors
  • API testing with Postman: Systematically test endpoints with valid and invalid payloads, expired JWTs, and edge-case parameters to confirm validation, authorization rules, and error responses behave correctly before release.

React vs Next.js

  • React: A client-side library for building UI, ideal for highly interactive apps.
    • Routing via react-router-dom.
    • Code-splitting: use React.lazy + Suspense for component-level lazy loading.
    • Example: lazy-loading pages or rarely-used components inside a SPA.
  • Next.js: A framework on top of React adding server-side rendering, static site generation, and file-based routing. Improves SEO and first-load performance.
    • Routing is automatic; each file in pages/ becomes a route.

Summary: React SPAs require explicit lazy-loading and routing setup. Next.js handles page-level code-splitting and routing automatically, with optional dynamic imports for components, giving a clean, scalable, and SEO-friendly architecture.

React State Batching

  • Key Point: React does not immediately apply state changes, so multiple calls using the same value can overwrite each other. Functional updates in React receive the previous state as an argument and return the new state, ensuring that updates are always based on the most recent pending state.
  • Solution: Use an updater function to reliably increment state: setCount(prev => prev + 1)

React useMemo

  • Key Point: useMemo caches the result of an expensive calculation and only recalculates when dependencies in the array change.
  • Use case: Avoid redundant work on re-renders when the computation depends on specific props or state.

import React, { useState, useMemo } from 'react';

const ExpensiveCalculationComponent: React.FC<{ numbers: number[] }> = ({ numbers }) => {
  // useMemo caches the result and only recalculates when 'numbers' changes
  const total = useMemo(() => {
    console.log("Calculating sum..."); // shows when calculation runs
    return numbers.reduce((acc, n) => acc + n, 0);
  }, [numbers]); // dependency array: only recalc if 'numbers' changes

  return <div>Total Sum: {total}</div>;
};

const App: React.FC = () => {
  const [count, setCount] = useState(0);
  const numbers = [1, 2, 3, 4, 5];

  return (
    <div>
      {/* ExpensiveCalculationComponent only recalculates total if 'numbers' changes */}
      <ExpensiveCalculationComponent numbers={numbers} />

      {/* Updating 'count' does NOT trigger recalculation of total */}
      <button onClick={() => setCount(count + 1)}>Increment: {count}</button>
    </div>
  );
};

export default App;

React useRef

  • Key Point: useRef stores a mutable value that persists across renders without causing a re-render when it changes.
  • Use case: Access DOM elements directly or store values that should not trigger UI updates. 

import React, { useState, useRef, useEffect } from 'react';

const FocusInputComponent: React.FC = () => {
  const inputRef = useRef(null); 
  // useRef persists across renders

  const [count, setCount] = useState(0);

  useEffect(() => {
    console.log("Component rendered");
  });

  const focusInput = () => {
    // Accessing DOM element directly via ref
    inputRef.current?.focus();
  };

  return (
    <div>
      <input ref={inputRef} placeholder="Click button to focus" />

      {/* Does NOT trigger re-render when changed */}
      <button onClick={focusInput}>Focus Input</button>

      {/* This DOES trigger re-render */}
      <button onClick={() => setCount(count + 1)}>
        Re-render: {count}
      </button>
    </div>
  );
};

export default FocusInputComponent;

React useReducer

  • Key Point: useReducer manages complex state logic using a reducer function instead of multiple useState calls.
  • Use case: When state transitions depend on previous state or involve multiple related values.

import React, { useReducer } from 'react';

type State = { count: number };

type Action =
  | { type: 'increment' }
  | { type: 'decrement' }
  | { type: 'reset' };

const reducer = (state: State, action: Action): State => {
  switch (action.type) {
    case 'increment':
      return { count: state.count + 1 };
    case 'decrement':
      return { count: state.count - 1 };
    case 'reset':
      return { count: 0 };
    default:
      return state;
  }
};

const Counter: React.FC = () => {
  const [state, dispatch] = useReducer(reducer, { count: 0 });
  // state = current state
  // dispatch = function to trigger state transitions

  return (
    <div>
      <p>Count: {state.count}</p>

      <button onClick={() => dispatch({ type: 'increment' })}>+</button>
      <button onClick={() => dispatch({ type: 'decrement' })}>-</button>
      <button onClick={() => dispatch({ type: 'reset' })}>Reset</button>
    </div>
  );
};

export default Counter;

useToggle

  • Key Point: Simplifies managing boolean state, allowing easy toggling between true and false.
  • Use case: Toggling modals, dropdowns, checkboxes, loading states, or any boolean flag.

import { useState, useCallback } from 'react';

const useToggle = (initialValue: boolean = false) => {
  const [value, setValue] = useState(initialValue);

  const toggle = useCallback((nextValue?: boolean) => {
    setValue(current => 
      typeof nextValue === 'boolean' ? nextValue : !current
    );
  }, []);

  return [value, toggle] as const;
};

// Usage example
const ToggleComponent: React.FC = () => {
  const [isOpen, toggleOpen] = useToggle(false);

  return (
    <div>
      <p>Modal is {isOpen ? 'Open' : 'Closed'}</p>
      <button onClick={() => toggleOpen()}>Toggle</button>
      <button onClick={() => toggleOpen(true)}>Open</button>
      <button onClick={() => toggleOpen(false)}>Close</button>
    </div>
  );
};

export default ToggleComponent;

useDebounce

  • Key Point: Delays invoking a callback until a specified delay has passed since the last change.
  • Use case: Avoid excessive API calls or side effects when users are typing in inputs or performing rapid actions.

import { useEffect } from 'react';
import useTimeout from './useTimeout'; // assumes a basic useTimeout hook exists

const useDebounce = (callback: () => void, delay: number, deps: any[] = []) => {
  const { set: setTimeoutHook, clear: clearTimeoutHook } = useTimeout(callback, delay);

  useEffect(() => {
    clearTimeoutHook();        // clear previous timeout on first run
    setTimeoutHook();          // restart timeout whenever deps change
  }, [...deps, setTimeoutHook, clearTimeoutHook]);
};

// Usage example
const DebounceComponent: React.FC = () => {
  const [query, setQuery] = useState('');

  useDebounce(() => {
    console.log('API call with query:', query);
  }, 1000, [query]);

  return (
    <input
      value={query}
      onChange={e => setQuery(e.target.value)}
      placeholder="Type to search..."
    />
  );
};

export default DebounceComponent;

JavaScript Core Concepts

var vs let vs const

Scope Differences

  • var is function-scoped.
  • let and const are block-scoped.
  • Use const by default to prevent reassignment (e.g., API URL, array/object reference).
  • Use let for variables that need reassignment (e.g., counters, loop indices).

function example() {
  if (true) {
    var a = 1;   // function-scoped
    let b = 2;   // block-scoped
    const c = 3; // block-scoped
  }

  console.log(a); // ✅ 1
  console.log(b); // ❌ ReferenceError
  console.log(c); // ❌ ReferenceError
}

Hoisting

Hoisting is JavaScript’s behavior of moving variable and function declarations to the top of their scope during the compilation phase, before code execution.

Variable Hoisting

  • var declarations are hoisted and initialized with undefined.
  • let and const are hoisted but remain in the Temporal Dead Zone (TDZ) until initialized, causing a ReferenceError if accessed before declaration.

console.log(a); // undefined
var a = 5;

console.log(b); // ReferenceError
let b = 10;

Function Hoisting

  • Function declarations are fully hoisted and can be called before they appear in the code.
  • Function expressions are not fully hoisted (they behave like variables).

sayHello(); // ✅ Works

function sayHello() {
  console.log("Hello");
}

greet(); // ❌ TypeError

var greet = function() {
  console.log("Hi");
};

Summary: During compilation, JavaScript hoists declarations to the top of their scope. var is initialized as undefined, let/const are inaccessible until initialized (TDZ), and function declarations are fully hoisted.

Pass by Value vs Pass by Reference

  • Everything in JavaScript is pass by value, but for objects the value is a reference
    • Pass by value: The function gets a copy of the variable’s value. Changes inside the function do not affect the original variable.
      • 
  let a = 10;       // Primitive (number) → stored by value
  let b = "Hi";     // Primitive (string) → stored by value
  
  let c = [1, 2];   // Array → reference type
  let d = c;        // d now references the same array as c
  
  d.push(3);        // Mutates the array referenced by both c and d
  console.log(c);   // [1, 2, 3] → c sees the change because arrays are reference types
  console.log(d);   // [1, 2, 3]
  
  d = [3, 4, 5];    // d now references a new array
  console.log(c);   // [1, 2, 3] → c is unchanged, still references the original array
  console.log(d);   // [3, 4, 5] → d now points to the new array

  // Object equality
  let x = [1, 2, 3];
  let y = [1, 2, 3];

  console.log(x === y);               // false → different references in memory
  console.log(x.toString() === y.toString()); // true → contents serialized as string match

for...in vs for...of

  • for...in iterates over the keys (property names) of an object — use for plain objects when you need keys
  • for...of iterates over the values of iterables (arrays, strings, Map, Set) — use for arrays and other iterables when you need values
      
    // for...in — iterates over object keys, use to access property values
    const user = { name: "Alice", age: 30 };
    for (const key in user) {
      console.log(user[key]); // "Alice", 30
    }

    // for...of — iterates over array values directly
    const colors = ["red", "green", "blue"];
    for (const color of colors) {
      console.log(color); // "red", "green", "blue"
    }

Backend & Architecture

What are microservices and distributed systems?

Microservices break an application into small, independently deployable services, each handling a single business function and communicating via APIs. They were created to address scaling, deployment, and maintainability challenges in monolithic systems. For example, in a JavaScript e-commerce platform, authentication, orders, payments, and inventory could run as separate Node.js services using PostgreSQL and Redis. If order traffic spikes, you can scale just the Orders service with multiple Docker containers instead of the whole application. Other distributed systems exist too. For example, a PostgreSQL cluster with a primary server for writes and read-only replicas to handle high query load, or a Redis caching cluster with data sharded across nodes to speed up access by keeping frequently requested values in memory, are distributed systems but not microservices because they aren't separate business services.

Message Brokers

Message brokers like RabbitMQ allow different parts of a distributed system to communicate asynchronously without blocking each other. This keeps the system responsive even if one service is slow and lets background tasks, such as sending emails or generating reports, run in parallel. I haven't used RabbitMQ professionally yet, but I've built hands-on familiarity independently—for example, in a local Node.js project where I used RabbitMQ to queue email notifications. When a user submitted a form, the app pushed a message to a queue instead of sending the email immediately. Given my experience with similar stacks and architectural patterns, I'm confident I could ramp up quickly and become productive.

SQL vs NoSQL

SQL is a relational database that stores data in structured tables with rows and columns, ideal for complex relationships and strict validation. For example, tracking customer orders, payments, and inventory where relationships and constraints matter. NoSQLis a non-relational database that stores data in documents, key-value pairs, or graphs, which is better for flexible or unstructured schemas. For example, storing user-generated content like social media posts, where some posts may have text, images, videos, or polls, or product reviews where different users include optional ratings, comments, or attachments—so each record can have different fields and structures without breaking the database. 

  // SQL example (relational)
  const usersTable = [
    { id: 1, name: "Alice", email: "alice@example.com" },
    { id: 2, name: "Bob", email: "bob@example.com" }
  ];

  // Query: get user by ID
  const user = usersTable.find(u => u.id === 1);
  console.log(user.name); // Alice

  // NoSQL example (document database)
  const usersCollection = {
    "user_1": { name: "Alice", email: "alice@example.com" },
    "user_2": { name: "Bob", email: "bob@example.com" }
  };

  // Query: get user by key
  console.log(usersCollection["user_1"].name); // Alice

GitHub Actions

GitHub Actions is a CI/CD platform built into GitHub that automates workflows like testing, building, and deploying code. Workflows are defined in YAML files and triggered on events such as pushes, pull requests, or scheduled jobs. This ensures consistent, repeatable pipelines and reduces manual deployment errors. 

# Example workflow: Trivia App Backend
# Trigger: push to main/master branch
# Tasks: test, build Docker image, push to ECR, deploy to ECS
# Secrets: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION, ECR_URI, ECS_CLUSTER, ECS_SERVICE, ECS_TASK

name: Backend Deploy

on:
  push:
    branches: [main, master]
    paths:
      - 'backend/**'
      - '.github/workflows/backend-deploy.yml'

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: trivia-master
    defaults:
      run:
        working-directory: backend

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: backend/package-lock.json

      - name: Install dependencies
        run: npm ci

      - name: Generate Prisma client
        run: npx prisma generate

      - name: Run tests
        run: npm run test

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ secrets.AWS_REGION || 'us-east-1' }}

      - name: Login to Amazon ECR
        uses: aws-actions/amazon-ecr-login@v2

      - name: Set image tag
        id: tag
        run: echo "SHORT_SHA=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT

      - name: Build and push Docker image
        env:
          ECR_URI: ${{ secrets.ECR_URI }}
          SHORT_SHA: ${{ steps.tag.outputs.SHORT_SHA }}
        run: |
          set -e
          docker buildx create --use --name buildx-builder 2>/dev/null || true
          docker buildx build --platform linux/amd64 \
            -f Dockerfile.prod \
            -t $ECR_URI:$SHORT_SHA \
            -t $ECR_URI:latest \
            --provenance=false \
            --push .

      - name: Deploy to ECS
        run: |
          set -e
          aws ecs update-service \
            --cluster ${{ secrets.ECS_CLUSTER }} \
            --service ${{ secrets.ECS_SERVICE }} \
            --force-new-deployment \
            --region ${{ secrets.AWS_REGION || 'us-east-1' }}

Key points illustrated by this workflow:

  • Automated testing ensures backend code works before deploying.
  • Docker builds reproducible, portable container images targeting linux/amd64.
  • Images are pushed to Amazon ECR and deployed to ECS using rolling updates.
  • Secrets and credentials are stored securely in GitHub.
  • Workflows are triggered only on relevant changes (backend code or workflow file).
Using GitHub Actions like this enables safe, consistent, and repeatable deployments from development to production.

React API Call Optimization

Optimizing API calls is key for dashboards or pages with multiple data-dependent panels.
  • Sequential await calls: Requests run one after another, making total load time slower.
  • Independent useEffect hooks: Each API call runs concurrently, but this approach becomes verbose and harder to maintain with many panels.
  • Promise.all() (recommended): Runs requests concurrently, waits for all to finish, and keeps code clean with async/await.
    
    import axios from 'axios';
    import { useEffect, useState } from 'react';
    
    const Dashboard = () => {
      const [panelA, setPanelA] = useState(null);
      const [panelB, setPanelB] = useState(null);
      const [panelC, setPanelC] = useState(null);
    
      useEffect(() => {
        const fetchData = async () => {
          try {
            const [resA, resB, resC] = await Promise.all([
              axios.get('/api/popA'),
              axios.get('/api/popB'),
              axios.get('/api/popC'),
            ]);
    
            setPanelA(resA.data);
            setPanelB(resB.data);
            setPanelC(resC.data);
          } catch (err) {
            console.error(err);
          }
        };
    
        fetchData();
      }, []);
    
      return (
        <div>
          <Panel data={panelA} />
          <Panel data={panelB} />
          <Panel data={panelC} />
        </div>
      );
    };

Key takeaway: Promise.all() is the most common and modern best practice for parallel API calls in React, providing concurrency, clear handling of results, and clean syntax.

Virtual DOM & Reconciliation

The Virtual DOM is an in-memory representation of the real DOM. React uses it to efficiently update the UI by minimizing slow, direct DOM manipulations.

When state or props change, React builds a new virtual DOM tree and compares it to the previous one using the reconciliation algorithm:

  • Only differences (diffs) between the old and new virtual DOM are applied to the real DOM.
  • React uses heuristics, like element types and key props, to determine which nodes to update, replace, or remove.

Key benefits:

  • Improved performance by reducing unnecessary DOM operations.
  • Predictable, declarative UI updates.
  • Efficient list updates when keys are used correctly.

In short, the Virtual DOM and reconciliation allow React apps to update quickly and efficiently without manual DOM handling.

React Security

React provides several built-in protections, but developers still need to follow best practices to keep applications secure.

  • XSS Protection: React automatically escapes content rendered in JSX, preventing most cross-site scripting attacks. Avoid using dangerouslySetInnerHTML with untrusted input, or sanitize it first.
  • Secure Data Handling: Never store sensitive data in the client if possible, and avoid exposing API keys or secrets in frontend code.

  • Dependencies: Keep React and other libraries up-to-date to mitigate known vulnerabilities.
  • Input Validation: Validate and sanitize user input before sending it to APIs or databases to reduce injection risks.

Following these practices ensures React apps remain robust against common security threats while leveraging React’s inherent protections.

Syncing React State Across Multiple Browser Tabs

When the same React app is open in multiple browser tabs or windows, each tab has its own memory and React state. To keep them in sync, you need an external mechanism for cross-tab communication.

  • LocalStorage + Storage Event: Write updates to localStorage and listen for the storage event in other tabs to update state accordingly.
  • Service Workers / BroadcastChannel API: Use BroadcastChannel to send messages between tabs for real-time updates without polling.
  • Server-Side Sync: Persist state in a backend (e.g., via WebSockets or REST API) and have all tabs subscribe or fetch updates, ensuring consistent global state.

Choice depends on complexity and real-time needs: localStorage + storage works for simple settings or counters, while BroadcastChannel or WebSockets are better for collaborative or real-time dashboards.

Prop Drilling

Prop drilling occurs when data is passed from a parent component down to deeply nested child components through multiple intermediate components that do not actually need the data.

  • Intermediate components must accept and forward props they don’t use.
  • This can make components harder to read, maintain, and refactor.
  • Common in large React applications with deeply nested trees.

Solutions: React Context, Redux, Zustand, or other global state management approaches to avoid passing props through every level.

Summary: Prop drilling is the unnecessary passing of props through multiple layers of components to reach a deeply nested child.

React Lifecycle & Hooks

Purpose of the Dependency Array in useEffect

The dependency array in useEffect tells React when to re-run the effect based on changes to specific values.

  • With dependency array: useEffect(() => { ... }, [dep1, dep2]) The effect runs only when one of the listed dependencies changes. This is useful for fetching data or updating state only when certain values change, preventing unnecessary re-renders or side effects.
  • Empty dependency array: useEffect(() => { ... }, []) Runs once on mount and not on subsequent renders. Ideal for initialization logic like fetching initial data or subscribing to events.
  • No dependency array: useEffect(() => { ... }) Runs on every render. Usually avoided unless the effect must react to every render, as it can hurt performance.

Summary: The dependency array controls the frequency and timing of effect execution, helping optimize performance and avoid unintended behavior.

Purpose of the return Statement in useEffect

The return statement in a useEffect hook defines a cleanup function. This function runs when the component unmounts or before the effect runs again (if dependencies change), allowing you to clean up subscriptions, timers, or event listeners to avoid memory leaks.

  • Example - Event Listener: Add a window resize listener and remove it on unmount. 

useEffect(() => {
  const handleResize = () => {
    console.log(window.innerWidth);
  };

  window.addEventListener('resize', handleResize);

  // Cleanup function
  return () => {
    window.removeEventListener('resize', handleResize);
  };
}, []); // Empty dependency array: runs once on mount

Summary: The return cleanup in useEffect ensures resources are properly released, preventing memory leaks or unintended behavior when components unmount or dependencies change.

Routing

Client-Side Routing vs Server-Side Routing

Client-Side Routing is handled in the browser using JavaScript. Instead of requesting a new HTML page from the server for every route change, the app dynamically updates the UI without a full page reload.

  • Used in Single Page Applications (SPAs) like React apps.
  • Navigation feels fast because only data changes, not the entire page.
  • Routing is managed by libraries like React Router.
  • Initial load may be slower because the full app bundle must download first.

Server-Side Routing is handled by the server. Each route change sends a new request to the server, which responds with a completely new HTML page.

  • Traditional multi-page applications (MPAs).
  • Each navigation triggers a full page reload.
  • SEO-friendly by default because HTML is rendered on the server.
  • Can feel slower due to full page refreshes.

Summary: Client-side routing prioritizes speed and smooth UX in SPAs, while server-side routing relies on full page reloads and is naturally SEO-friendly.

What is a Closure?

A closure occurs when a function "remembers" variables from its outer (lexical) scope even after that outer function has finished executing.

In JavaScript, functions retain access to the variables that were in scope at the time they were created. This allows data to persist in memory without being globally accessible.

Practical Example – Private Counter


function createCounter() {
  let count = 0;

  return function () {
    count++;
    console.log(count);
  };
}

const counter = createCounter();

counter(); // 1
counter(); // 2
counter(); // 3

Even though createCounter() has finished running, the inner function still has access to count. That variable is preserved via closure.

Demonstrating "Live References"


function outer() {
  let value = 10;

  return {
    increment: () => value++,
    getValue: () => value
  };
}

const example = outer();

console.log(example.getValue()); // 10
example.increment();
console.log(example.getValue()); // 11

Closures hold a live reference to the variable, not a snapshot. When value changes, every function inside the closure sees the updated value.

Common Use Cases:

  • Data privacy (encapsulation)
  • Factory functions
  • Event handlers
  • React hooks and state management

Summary: A closure is a function that retains access to its lexical scope, allowing variables to persist beyond their original execution context.

What is a Pure Function?

A pure function is a function that:

  • Always returns the same output for the same input.
  • Has no side effects (it does not modify external variables, make API calls, or change application state).
  • Important in React and Redux (reducers must be pure)

JavaScript Event Loop

The Event Loop is the mechanism that allows JavaScript (which is single-threaded) to handle asynchronous operations like timers, API calls, and promises without blocking the main thread.

How It Works

  • JavaScript executes code on the Call Stack (one task at a time).
  • Async operations (e.g., setTimeout, fetch, promises) are handled by the browser/Web APIs.
  • When they complete, their callbacks are placed in the Task Queue (or Microtask Queue for promises).
  • The Event Loop continuously checks: if the Call Stack is empty, it pushes queued tasks onto it.

Important: Microtasks (e.g., Promises) run before regular tasks (e.g., setTimeout).

Summary: The Event Loop coordinates the Call Stack and Task Queues to enable non-blocking, asynchronous behavior in JavaScript.

Parent–Child Data Sharing (Callback Pattern)

In React, data flows downward via props. To allow a child component to send data back to a parent, we use a callback function passed from the parent to the child.

How It Works

  • The parent defines a function that updates its state.
  • The parent passes that function to the child as a prop.
  • The child calls the function and passes data as an argument.
  • The parent receives the data and updates its state.

Example


// Parent Component
function Parent() {
  const [message, setMessage] = React.useState("");

  const handleMessage = (data) => {
    setMessage(data);
  };

  return (
    <div>
      <Child onSendMessage={handleMessage} />
      <p>Message from child: {message}</p>
    </div>
  );
}

// Child Component
function Child({ onSendMessage }) {
  return (
    <button onClick={() => onSendMessage("Hello Parent!")}>
      Send Message
    </button>
  );
}

Summary: The callback pattern enables child-to-parent communication while maintaining React’s one-way data flow.

Double (==) vs. Triple (===) Equality in JavaScript

Double equality (==) checks for value equality after coercing types. It can lead to unexpected results because JavaScript converts one or both values to a common type before comparing.

  • 5 == "5" → true (string is coerced to number)
  • null == undefined → true
  • 0 == false → true

Triple equality (===) checks for value and type equality. No type coercion occurs, making it safer and more predictable.

  • 5 === "5" → false
  • null === undefined → false
  • 0 === false → false

Best practice: Use === (triple equality) unless you explicitly need type coercion.

Null vs Undefined in JavaScript

undefined indicates a variable has been declared but not assigned a value. null is an assignment value that represents “no value” or “empty”.

  • undefined: automatically assigned by JavaScript
  • null: explicitly assigned by the developer to indicate intentional absence

// undefined example
let a;
console.log(a); // undefined

// null example
let b = null;
console.log(b); // null

// Practical use
function getUser(id) {
  if (idExists(id)) {
    return { name: "Alice" };
  } else {
    return null; // explicitly indicate no user found
  }
}

let result = getUser(42);
if (result === null) {
  console.log("User not found");
}

Key point: undefined is for “missing value by default”, null is for “deliberate empty value”. Use null to explicitly clear variables or indicate no result; undefined usually signals an uninitialized variable.

REST Verbs

REST (Representational State Transfer) APIs use standard HTTP verbs to indicate the type of operation to perform on resources. Each verb has a conventional purpose and standard response codes:

  • GET – Retrieve data from the server. Should not modify any state.
    Example: fetch a list of users.
    Typical Response: 200 OK, 404 Not Found
  • POST – Create a new resource on the server.
    Example: add a new user to the database.
    Typical Response: 201 Created, 400 Bad Request
  • PUT – Update an existing resource entirely. Idempotent.
    Example: update all fields of a user.
    Typical Response: 200 OK, 204 No Content, 404 Not Found
  • PATCH – Partially update an existing resource.
    Example: update a user's email address.
    Typical Response: 200 OK, 204 No Content, 404 Not Found
  • DELETE – Remove a resource from the server.
    Example: delete a user account.
    Typical Response: 200 OK, 204 No Content, 404 Not Found

Summary: Using the correct REST verb and response codes ensures predictable behavior, proper API semantics, and easier client integration.

Class Components vs Functional Components

React supports two ways to define components: class components and functional components. Key differences and benefits:

  • Syntax:
    • Class components: use ES6 classes, extend React.Component, require a render() method.
    • Functional components: use plain functions, return JSX directly.
  • State and Lifecycle:
    • Class components manage state via this.state and lifecycle methods like componentDidMount.
    • Functional components use useState, useEffect, and other hooks for state and side effects.
  • Readability and Conciseness:
    • Functional components are shorter and easier to read, with less boilerplate.
  • Performance:
    • Initially class components had slightly more overhead, but now both are similar; functional components can leverage React.memo and hooks for optimization.
  • Hooks:
    • Functional components can use hooks (useState, useEffect, useContext, etc.), which provide more flexibility and composability than lifecycle methods.

Summary: Functional components are now preferred due to simplicity, less boilerplate, easier testing, better composability with hooks, and modern React performance patterns. Class components are still valid but increasingly rare in modern React codebases.

Redis

Redis is an in-memory, multi-model database known for sub-millisecond latency. It combines fast memory reads/writes with optional disk persistence for durability.

  • Data is stored as key-value pairs, with structures such as strings, lists, hashes, sets, streams, and more.
  • Commonly used as a cache to accelerate relational databases or APIs, but can also serve as a primary database.
  • Modular extensions support graphs, JSON, full-text search, time series, and AI workloads.
  • Commands are simple and intuitive (GET, SET, HSET, LPUSH, etc.), enabling rapid development.
  • Redis Enterprise Cloud offers a free tier to explore features safely and efficiently.

Summary: Redis provides extremely fast, flexible, and durable data storage suitable for caching, real-time analytics, messaging, and modern cloud applications.

Pagination

Pagination is the practice of dividing large datasets into smaller, manageable chunks to improve performance and user experience. Instead of loading all data at once, the client requests only a subset of records per page or batch.

  • Types: Offset-based (skip & limit), cursor-based (using a pointer to the last item), or keyset pagination.
  • Use cases: Displaying user lists, feed items, search results, or API responses.
  • Benefits: Reduces server load, speeds up response times, improves UI rendering, and minimizes memory consumption.
  • Implementation: Backend endpoints usually accept query parameters like page, limit, or cursor to return the corresponding subset of data.

Summary: Pagination ensures efficient data fetching and smooth user experience when handling large datasets in applications or APIs.

React Window

React Window is a library for efficiently rendering large lists or grids in React by only rendering the visible portion of the data in the viewport. This technique, known as virtualization, drastically improves performance and reduces memory usage for huge datasets, making it ideal for tables, feeds, or any long scrollable lists.

  • How it works: React Window calculates which items are visible based on scroll position and renders only those items. Off-screen items are not mounted, reducing DOM nodes.
  • When to use: Any scenario with a very large list (hundreds or thousands of items) where rendering all elements at once would slow down the app.

import React from 'react';
import { FixedSizeList as List } from 'react-window';

interface User {
  id: number;
  name: string;
}

const users: User[] = Array.from({ length: 10000 }, (_, i) => ({
  id: i,
  name: `User ${i + 1}`
}));

const Row = ({ index, style }: { index: number; style: React.CSSProperties }) => (
  <div style={style}>
    {users[index].name}
  </div>
);

const App: React.FC = () => (
  <List
    height={500}             // Height of the container
    itemCount={users.length} // Total number of items
    itemSize={35}            // Height of each row
    width={'100%'}           // Container width
  >
    {Row}
  </List>
);

export default App;

Summary: React Window enables performant rendering of very large lists by virtualizing off-screen items, making your React app smoother and more responsive without sacrificing functionality.

Updating Objects in useState with Spread Notation

When managing state that is an object in React, always use the functional form of the setter and spread the previous state. This ensures only the intended property is updated while preserving the rest of the object.

  • Use prevState => ({ ...prevState, property: newValue }) to update a single property.
  • For forms with multiple inputs, a generic handleChange function can leverage e.target.name to dynamically update the correct property. 

const [formData, setFormData] = useState({
  name: "",
  email: "",
  password: ""
});

const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  setFormData(prev => ({
    ...prev,
    [e.target.name]: e.target.value
  }));
};

// Usage in JSX:
<input
  type="text"
  name="name"
  value={formData.name}
  onChange={handleChange}
/>
<input
  type="email"
  name="email"
  value={formData.email}
  onChange={handleChange}
/>

Summary: Spread syntax combined with [e.target.name]: e.target.value allows concise, dynamic updates to object state, keeping all other properties intact.

Derived State in React

In React, avoid creating extra useState hooks for values that can be computed from existing state or props. Instead, derive them dynamically to reduce complexity, prevent bugs, and keep your state minimal.

  • Example: Compute fullName from firstName and lastName instead of storing it in state.
  • Example: Calculate totalPrice as quantity * unitPrice instead of keeping a separate state for total.

// Instead of:
const [fullName, setFullName] = useState(firstName + " " + lastName);

// Use derived value:
const fullName = `${firstName} ${lastName}`;

// Instead of:
const [totalPrice, setTotalPrice] = useState(quantity * unitPrice);

// Use derived value:
const totalPrice = quantity * unitPrice;

Summary: Derived values computed from props or existing state simplify your component logic, reduce unnecessary re-renders, and prevent stale or inconsistent state.

Organizing React Projects

A well-structured React project improves maintainability, discoverability, and scalability. Common folder conventions:

  • components/ – Reusable UI components, e.g., Button.tsx, Modal.tsx.
  • pages/ – Page-level or route-level components (especially in Next.js), e.g., Home.tsx, Profile.tsx.
  • hooks/ – Custom React hooks, e.g., useFetch.ts, useForm.ts.
  • utils/ – Utility functions/helpers, e.g., formatDate.ts, calculateTax.ts.
  • assets/ – Static assets like images, icons, fonts, e.g., logo.svg, background.jpg.
  • data/ – Mock data, constants, or JSON fixtures, e.g., countries.json, mockUsers.ts.
  • contexts/ – React Context providers, e.g., AuthContext.tsx, ThemeContext.tsx.
  • services/ – API calls or business logic, e.g., apiClient.ts, userService.ts.
  • styles/ – Global or shared CSS/SCSS, e.g., variables.scss, globals.css.
  • tests/ – Unit/integration test files, sometimes colocated with components, e.g., Button.test.tsx.

Example folder structure:

│  ├─ images/
│  └─ icons/
├─ components/
│  ├─ Button.tsx

│     ├─ Header.tsx

├─ pages/
│  ├─ Home.tsx
│  └─ Profile.tsx
├─ hooks/
│  ├─ useFetch.ts
│  └─ useForm.ts

│  ├─ formatDate.ts
│  └─ calculateTax.ts
├─ contexts/

├─ services/

└─ tests/

   └─ Header.test.tsx

Summary: Organizing by feature or by type (components, hooks, utils, etc.) keeps a React project maintainable and scalable, especially as the codebase grows.

Cursor IDE Integration with MCP Servers

Cursor IDE can be integrated with various MCP (Managed Cloud Platform) services to streamline developer workflows, monitor issues, and automate ticket management.

  • Figma MCP server: Access design assets, components, and specs directly within the IDE. Developers can preview and reference design files without leaving their coding environment.
  • Stripe MCP server: Query payment events, view logs, and debug transactions. Allows developers to quickly inspect API responses and integrate payments seamlessly.
  • Sentry monitoring tool: MCP integration lets developers query errors and issues reported by Sentry directly from Cursor. Provides quick navigation to stack traces, impacted users, and context for debugging.
  • Atlassian or GitHub MCP server: Automate ticket and issue tracking by pulling tasks directly into the IDE. Developers can link commits, reference tickets, and even trigger fixes or status updates without switching tools.
  • GitHub Actions MCP server: Integrate build/test pipelines, see workflow status, rerun jobs, and debug failing pipelines from Cursor.
  • MongoDB Atlas MCP server: Query collections, inspect documents, and run simple CRUD queries.
  • Redis MCP server: Check key/values or cache metrics in real time.

Summary: Integrating Cursor IDE with MCP servers and developer tools centralizes workflows, reduces context switching, and accelerates feature development and debugging.

AWS Overview

AWS (Amazon Web Services) started in 2006 with just 3 services (S3, EC2, SQS) and has grown to 200+ services. It offers flexible compute, storage, database, container, ML, IoT, and DevOps solutions, enabling developers to launch and scale applications efficiently.

  • Compute: EC2 (VMs), Lambda (serverless), Elastic Beanstalk & Lightsail (managed deployment), Fargate & App Runner (containers), Outposts & Snow (on-prem/edge).
  • Storage: S3 (object), Glacier (archival), EBS (block), EFS (file), plus databases: DynamoDB, RDS, Aurora, Neptune, DocumentDB, ElastiCache, Timestream, QLDB.
  • Containers: ECR (registry), ECS & EKS (orchestration), Fargate (serverless containers).
  • Analytics & ML: Redshift (warehouse), Lake Formation (data lake), Kinesis/MSK (streaming), EMR (big data), Glue (ETL), SageMaker (ML), Rekognition, Lex, DeepRacer.
  • IoT & Robotics: IoT Core, RoboMaker, Ground Station, Braket (quantum computing).
  • Security & Identity: IAM (permissions), Cognito (user auth).
  • Notifications & Messaging: SNS, SES.
  • Automation & DevOps: CloudFormation (IaC), Amplify (frontend SDKs).
  • Cost Management: Cost Explorer, Budgets.

Core concepts: most cloud services are built on a few fundamental resources:

  • VMs: Provide CPU, RAM, and disk. Base layer for all other services, fully managed or unmanaged.
  • Object Storage: S3-style storage abstracts replication and scaling for files.
  • Databases: Managed relational (RDS) or non-relational (DynamoDB) storage simplifies operations.
  • Functions-as-a-Service: Lambda-style serverless compute for event-driven workloads and APIs.

Additional notes:

  • Most services wrap open-source software to simplify development.
  • Observability tools (logging, monitoring, alerts) improve reliability and performance.
  • Services can be regional (VMs, storage) or global (CDNs, global DBs), impacting latency and cost.
  • Hands-on experience is essential; understanding these core categories makes switching cloud providers easier.

Summary: AWS provides a comprehensive “grocery store” of tools—from compute and storage to ML and IoT—allowing developers to deploy, scale, and manage applications with flexible, cloud-native solutions.

Supabase

Supabase is an open-source backend alternative to Firebase, providing database, file storage, and serverless Edge functions. It connects easily to front-end frameworks like React, React Native, and Flutter via client SDKs.

Key features:

  • Postgres database: with a visual UI, automatic REST/GraphQL APIs, real-time updates, and row-level security.
  • Built-in authentication: integrated with the database.
  • Supports triggers, stored procedures, and serverless Edge functions for custom backend logic.
  • Can be self-hosted with Docker or used via a managed service with a free tier.
  • Simplifies full-stack development: no raw SQL required, real-time auth/state listeners, and globally distributed functions.

Summary: Supabase is designed for developers to build authenticated, scalable full-stack apps quickly without vendor lock-in.
