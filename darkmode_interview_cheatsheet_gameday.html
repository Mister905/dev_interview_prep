Organizing API Projects

I organize Node.js API projects by feature or domain so related endpoints, services, models, DTOs, and tests are grouped together. Controllers handle requests and delegate logic to services, models define database schemas or validation rules, middleware manages cross-cutting concerns like authentication or logging, utils provide helpers, and tests validate functionality. This structure keeps the API modular, maintainable, and predictable, making it easier to extend and troubleshoot.
  • Feature-based folders: Group all resources per domain, e.g., users/, posts/, orders/.
  • Controllers: Handle incoming requests and orchestrate calls to services.
  • Services: Encapsulate business logic and data operations.
  • Entities/Models: Define database schemas or ORM models.
  • DTOs (Data Transfer Objects): Define request/response shapes and provide a place for validation rules.
  • Middleware: Manage authentication, logging, or other cross-cutting concerns.
  • Tests: Unit and integration tests colocated for each feature.

Authentication

The system I’m most familiar with uses JWTs for authentication. When a user logs in, the backend issues a short-lived access token and a longer-lived refresh token. Each API request checks the access token to make sure it’s valid and that the user has the right role. Middleware and route guards enforce workflow rules, like protecting endpoints, restricting frontend routes, and preventing users from skipping approval steps. Critical checks, like business rules and data integrity, always run on the server so it remains the source of truth. Refresh tokens let users stay logged in without repeatedly signing in, and the backend can revoke them if needed to keep access secure.

On the frontend, the refresh flow is usually handled automatically by a centralized HTTP client or interceptor. It watches for expired tokens or 401 responses and triggers a call to the refresh endpoint before retrying the request. This way, users rarely notice the refresh; the session feels continuous.

      },

REST Verbs

REST APIs use standard HTTP verbs to operate on resources. Each verb has a conventional purpose:

  • GET – Retrieve data without modifying state.
    • Example: fetch users
  • POST – Create a new resource.
    • Example: add a user
  • PUT – Replace an existing resource entirely.
    • Example: update all user fields
  • PATCH – Partially update a resource.
    • Example: change a user’s email
  • DELETE – Remove a resource.
    • Example: delete a user

Summary: Using the correct verb ensures predictable behavior and easier client integration.

Webhooks

Webhooks let one app notify another when an event happens, replacing polling with push updates.

  • How it works:
    1. Register a URL endpoint with a service (e.g., GitHub, Stripe, Slack).
    2. The service detects an event (e.g., new commit, payment received).
    3. The service sends a POST request with the event payload to your URL.
    4. Your app processes the event (async if needed) and responds 200 OK.
  • Use cases: Payment notifications, CI/CD triggers, chat integrations, real-time syncing, event-driven workflows.
  • Best practices: Verify request authenticity (HMAC), handle duplicates with idempotency, respond quickly.

Summary: Webhooks enable push-based communication, letting servers initiate updates rather than clients polling.

Redis

Redis is an in-memory, multi-model database known for sub-millisecond latency. It combines fast memory reads/writes with optional disk persistence for durability.

  • Data is stored as key-value pairs, with structures such as strings, lists, hashes, sets, streams, and more.
  • Commonly used as a cache to accelerate relational databases or APIs, but can also serve as a primary database.
  • Modular extensions support graphs, JSON, full-text search, time series, and AI workloads.
  • Commands are simple and intuitive (GET, SET, HSET, LPUSH, etc.), enabling rapid development.
  • Redis Enterprise Cloud offers a free tier to explore features safely and efficiently.

Summary: Redis provides extremely fast, flexible, and durable data storage suitable for caching, real-time analytics, messaging, and modern cloud applications.

Pagination

Pagination is the practice of dividing large datasets into smaller, manageable chunks to improve performance and user experience. Instead of loading all data at once, the client requests only a subset of records per page or batch.

  • Types: Offset-based (skip & limit), cursor-based (using a pointer to the last item), or keyset pagination.
  • Use cases: Displaying user lists, feed items, search results, or API responses.
  • Benefits: Reduces server load, speeds up response times, improves UI rendering, and minimizes memory consumption.
  • Implementation: Backend endpoints usually accept query parameters like page, limit, or cursor to return the corresponding subset of data.

Summary: Pagination ensures efficient data fetching and smooth user experience when handling large datasets in applications or APIs.

Prop Drilling

Prop drilling occurs when data is passed from a parent component down to deeply nested child components through multiple intermediate components that do not actually need the data.

  • Intermediate components must accept and forward props they don’t use.
  • This can make components harder to read, maintain, and refactor.
  • Common in large React applications with deeply nested trees.

Solutions: React Context, Redux, Zustand, or other global state management approaches to avoid passing props through every level.

Summary: Prop drilling is the unnecessary passing of props through multiple layers of components to reach a deeply nested child.

React Context

React Context is a built-in feature that lets you share data across components without passing props manually at every level, solving the "prop drilling" problem.

  • Use case: Global state like themes, authentication status, or user preferences.
  • Benefits: Simplifies state management for data that many components need, reduces repetitive prop passing, and keeps components cleaner.
  • Example: <ThemeContext.Provider value={theme}>…</ThemeContext.Provider>

Parent–Child Data Sharing (Callback Pattern)

In React, data flows downward via props. To allow a child component to send data back to a parent, we use a callback function passed from the parent to the child.

How It Works

  • The parent defines a function that updates its state.
  • The parent passes that function to the child as a prop.
  • The child calls the function and passes data as an argument.
  • The parent receives the data and updates its state.

Example


// Parent Component
function Parent() {
  const [message, setMessage] = React.useState("");

  const handleMessage = (data) => {
    setMessage(data);
  };

  return (
    <div>
      <Child onSendMessage={handleMessage} />
      <p>Message from child: {message}</p>
    </div>
  );
}

// Child Component
function Child({ onSendMessage }) {
  return (
    <button onClick={() => onSendMessage("Hello Parent!")}>
      Send Message
    </button>
  );
}

React API Call Optimization

Optimizing API calls is key for dashboards or pages with multiple data-dependent panels.
  • Sequential await calls: Requests run one after another, making total load time slower.
  • Independent useEffect hooks: Each API call runs concurrently, but this approach becomes verbose and harder to maintain with many panels.
  • Promise.all() (recommended): Runs requests concurrently, waits for all to finish, and keeps code clean with async/await.
    
    import axios from 'axios';
    import { useEffect, useState } from 'react';
    
    const Dashboard = () => {
      const [panelA, setPanelA] = useState(null);
      const [panelB, setPanelB] = useState(null);
      const [panelC, setPanelC] = useState(null);
    
      useEffect(() => {
        const fetchData = async () => {
          try {
            const [resA, resB, resC] = await Promise.all([
              axios.get('/api/popA'),
              axios.get('/api/popB'),
              axios.get('/api/popC'),
            ]);
    
            setPanelA(resA.data);
            setPanelB(resB.data);
            setPanelC(resC.data);
          } catch (err) {
            console.error(err);
          }
        };
    
        fetchData();
      }, []);
    
      return (
        <div>
          <Panel data={panelA} />
          <Panel data={panelB} />
          <Panel data={panelC} />
        </div>
      );
    };

Key takeaway: Promise.all() is the most common and modern best practice for parallel API calls in React, providing concurrency, clear handling of results, and clean syntax.

Virtual DOM & Reconciliation

The Virtual DOM is an in-memory representation of the real DOM. React uses it to efficiently update the UI by minimizing slow, direct DOM manipulations.

When state or props change, React builds a new virtual DOM tree and compares it to the previous one using the reconciliation algorithm:

  • Only differences (diffs) between the old and new virtual DOM are applied to the real DOM.
  • React uses heuristics, like element types and key props, to determine which nodes to update, replace, or remove.

Key benefits:

  • Improved performance by reducing unnecessary DOM operations.
  • Predictable, declarative UI updates.
  • Efficient list updates when keys are used correctly.

In short, the Virtual DOM and reconciliation allow React apps to update quickly and efficiently without manual DOM handling.

Hoisting

Hoisting is JavaScript’s behavior of moving variable and function declarations to the top of their scope during the compilation phase, before code execution.

Variable Hoisting

  • var declarations are hoisted and initialized with undefined.
  • let and const are hoisted but remain in the Temporal Dead Zone (TDZ) until initialized, causing a ReferenceError if accessed before declaration.

console.log(a); // undefined
var a = 5;

console.log(b); // ReferenceError
let b = 10;

Function Hoisting

  • Function declarations are fully hoisted and can be called before they appear in the code.
  • Function expressions are not fully hoisted (they behave like variables).

sayHello(); // ✅ Works

function sayHello() {
  console.log("Hello");
}

greet(); // ❌ TypeError

var greet = function() {
  console.log("Hi");
};

Summary: During compilation, JavaScript hoists declarations to the top of their scope. var is initialized as undefined, let/const are inaccessible until initialized (TDZ), and function declarations are fully hoisted.

Pass by Value vs Pass by Reference

  • Everything in JavaScript is pass by value, but for objects the value is a reference
    • Pass by value: The function gets a copy of the variable’s value. Changes inside the function do not affect the original variable.
      •

  let b = "Hi";     // Primitive (string) → stored by value
  
  let c = [1, 2];   // Array → reference type

What is a Closure?

A closure occurs when a function "remembers" variables from its outer (lexical) scope even after that outer function has finished executing.

In JavaScript, functions retain access to the variables that were in scope at the time they were created. This allows data to persist in memory without being globally accessible.

Practical Example – Private Counter


function createCounter() {
  let count = 0;

  return function () {
    count++;
    console.log(count);
  };
}

const counter = createCounter();

counter(); // 1
counter(); // 2
counter(); // 3

Even though createCounter() has finished running, the inner function still has access to count. That variable is preserved via closure.

Demonstrating "Live References"


function outer() {
  let value = 10;

  return {
    increment: () => value++,
    getValue: () => value
  };
}

const example = outer();

console.log(example.getValue()); // 10
example.increment();
console.log(example.getValue()); // 11

Closures hold a live reference to the variable, not a snapshot. When value changes, every function inside the closure sees the updated value.

Common Use Cases:

  • Data privacy (encapsulation)
  • Factory functions
  • Event handlers
  • React hooks and state management

Summary: A closure is a function that retains access to its lexical scope, allowing variables to persist beyond their original execution context.

JavaScript Event Loop

The Event Loop is the mechanism that allows JavaScript (which is single-threaded) to handle asynchronous operations like timers, API calls, and promises without blocking the main thread.

How It Works

  • JavaScript executes code on the Call Stack (one task at a time).
  • Async operations (e.g., setTimeout, fetch, promises) are handled by the browser/Web APIs.
  • When they complete, their callbacks are placed in the Task Queue (or Microtask Queue for promises).
  • The Event Loop continuously checks: if the Call Stack is empty, it pushes queued tasks onto it.

Important: Microtasks (e.g., Promises) run before regular tasks (e.g., setTimeout).

Summary: The Event Loop coordinates the Call Stack and Task Queues to enable non-blocking, asynchronous behavior in JavaScript.

Double (==) vs. Triple (===) Equality in JavaScript

Double equality (==) checks for value equality after coercing types. It can lead to unexpected results because JavaScript converts one or both values to a common type before comparing.

  • 5 == "5" → true (string is coerced to number)
  • null == undefined → true
  • 0 == false → true

Triple equality (===) checks for value and type equality. No type coercion occurs, making it safer and more predictable.

  • 5 === "5" → false
  • null === undefined → false
  • 0 === false → false

Best practice: Use === (triple equality) unless you explicitly need type coercion.

Kubernetes

Kubernetes orchestrates containers at scale. It manages deployment, scaling, load balancing, and self-healing of applications across multiple servers. Developers build and test containers with Docker, push images to a registry, and Kubernetes deploys them as pods, ensuring the desired number of replicas run, restarting failed containers, distributing traffic, and scaling based on demand. 

  # View pods in the cluster
  kubectl get pods

  # Scale a deployment
  kubectl scale deployment my-app --replicas=3

  # View logs from a pod
  kubectl logs 

  # Execute a command inside a container
  kubectl exec -it  -- /bin/bash

Kubernetes and Docker are commonly used together: Docker creates and tests containers, while Kubernetes manages their deployment, scaling, and health. Organizations often pair this with Terraform to provision the underlying infrastructure (VMs, networking, storage) before Kubernetes deploys applications—Terraform builds the house, Kubernetes manages the inhabitants.

NestJS

NestJS is a Node.js framework for scalable, maintainable server-side apps using TypeScript, decorators, and dependency injection.
  • Modules: Organize the app; declare controllers/providers; export providers to share across modules.
  • Providers: Business logic classes (@Injectable()), injected into controllers or other providers.
  • Controllers: Handle requests, delegate logic to providers; use decorators @Get(), @Post(), etc.
  • Dependency Injection: Providers auto-injected, enabling clean separation of concerns.
  • Decorators: @Module(), @Injectable(), @Controller(), @Get()/@Post(), @Inject()
  • Optional: Pipes (validate/transform input), Guards (auth), Interceptors (logging, modify responses)
  • Bootstrap: NestFactory.create(AppModule) → listen on port, initializes modules/providers/controllers.

Focus on modular structure, DI, and decorators for maintainable, testable backend code.

Git Commands & Concepts

We follow essential Git practices to manage source code, coordinate features, and maintain a clean history. Day-to-day operations include cloning repositories, syncing with remotes, managing branches, and carefully rebasing or merging code to ensure a linear, understandable history. Force operations are used sparingly with safety flags, and reset commands allow quick recovery from mistakes. 

  # Clone a remote repository
  git clone https://github.com/username/repo.git

  # Fetch latest changes from remote without touching your branch
  git fetch origin
  # Example: You're on feature/login and want to see what changed on master
  git log HEAD..origin/master --oneline --graph --decorate

  # Pull latest changes and merge into current branch
  git pull origin master
  # Automatically merges remote master into your branch. May create a merge commit.

  # Push changes to remote; force only with lease for safety
  git push origin feature-branch --force-with-lease

  # Create a new branch and switch to it
  git checkout -b new-feature

  # Merge another branch into your current branch
  git merge master
  # Preserves history with a merge commit, combining changes from both branches

  # Hard reset to a specific commit (dangerous; discards uncommitted changes)
  git reset --hard 

  # List branches and highlight current branch
  git branch -v

  # View commit history in a compact graph
  git log --oneline --graph --decorate

  # Stage changes
  git add 

  # Commit staged changes with a descriptive message
  git commit -m "Add login form"

  # Revert a specific commit safely
  git revert 

  # -------------------------
  # Rebase & squash workflow
  # -------------------------

  ## 1. Start a feature branch
  git checkout -b feature/login
  # Make multiple commits while developing locally

  ## 2. Keep your feature branch up-to-date with master
  git fetch origin
  git rebase origin/master
  # Replays your feature commits on top of the latest master
  # Creates a linear history without unnecessary merge commits

  ## 3. Clean up your commits before merging
  git rebase -i origin/master
  # Interactive rebase lets you squash/fixup small or trivial commits
  # Example: current commits
  a1b2c3 Add login form
  d4e5f6 Fix typo in login form
  g7h8i9 Adjust button styling

  # After interactive rebase:
  pick a1b2c3 Add login form
  squash d4e5f6 Fix typo in login form
  squash g7h8i9 Adjust button styling
  # Result: one clean commit
  Add login form

  ## 4. Test your branch after rebase
  # Ensure all functionality works with latest master changes

  ## 5. Push your branch
  # Use force-with-lease if branch was pushed before
  git push --force-with-lease origin feature/login

  ## 6. Open a pull request
  # PR now has a clean, linear history ready to merge into master

Best practices: Use descriptive commit messages, rebase feature branches before merging, avoid unnecessary force pushes, and maintain a clean, readable history. Tags mark releases, and CI/CD pipelines integrate with Git to trigger automated tests and deployments.

GitHub Actions

GitHub Actions is a CI/CD platform built into GitHub that automates workflows like testing, building, and deploying code. Workflows are defined in YAML files and triggered on events such as pushes, pull requests, or scheduled jobs. This ensures consistent, repeatable pipelines and reduces manual deployment errors. 

# Example workflow: Trivia App Backend
# Trigger: push to main/master branch
# Tasks: test, build Docker image, push to ECR, deploy to ECS
# Secrets: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION, ECR_URI, ECS_CLUSTER, ECS_SERVICE, ECS_TASK

name: Backend Deploy

on:
  push:
    branches: [main, master]
    paths:
      - 'backend/**'
      - '.github/workflows/backend-deploy.yml'

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: trivia-master
    defaults:
      run:
        working-directory: backend

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: backend/package-lock.json

      - name: Install dependencies
        run: npm ci

      - name: Generate Prisma client
        run: npx prisma generate

      - name: Run tests
        run: npm run test

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ secrets.AWS_REGION || 'us-east-1' }}

      - name: Login to Amazon ECR
        uses: aws-actions/amazon-ecr-login@v2

      - name: Set image tag
        id: tag
        run: echo "SHORT_SHA=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT

      - name: Build and push Docker image
        env:
          ECR_URI: ${{ secrets.ECR_URI }}
          SHORT_SHA: ${{ steps.tag.outputs.SHORT_SHA }}
        run: |
          set -e
          docker buildx create --use --name buildx-builder 2>/dev/null || true
          docker buildx build --platform linux/amd64 \
            -f Dockerfile.prod \
            -t $ECR_URI:$SHORT_SHA \
            -t $ECR_URI:latest \
            --provenance=false \
            --push .

      - name: Deploy to ECS
        run: |
          set -e
          aws ecs update-service \
            --cluster ${{ secrets.ECS_CLUSTER }} \
            --service ${{ secrets.ECS_SERVICE }} \
            --force-new-deployment \
            --region ${{ secrets.AWS_REGION || 'us-east-1' }}

Key points illustrated by this workflow:

  • Automated testing ensures backend code works before deploying.
  • Docker builds reproducible, portable container images targeting linux/amd64.
  • Images are pushed to Amazon ECR and deployed to ECS using rolling updates.
  • Secrets and credentials are stored securely in GitHub.
  • Workflows are triggered only on relevant changes (backend code or workflow file).
Using GitHub Actions like this enables safe, consistent, and repeatable deployments from development to production.

Dependencies vs DevDependencies

In a Node.js project, dependencies and devDependencies distinguish between packages needed at runtime versus only during development.

  • Dependencies: Required for the app to run in production. These are installed when someone runs npm install in a production environment.
    Common examples: react, react-dom, express, axios
  • DevDependencies: Needed only during development, testing, or building the app. These are not installed in production unless explicitly requested.
    Common examples: typescript, eslint, jest, webpack, vite

Use npm install package-name for dependencies and npm install package-name --save-dev for devDependencies to keep your production bundle lean.

npm vs npx

Both npm and npx are tools that come with Node.js, but they serve different purposes.

  • npm (Node Package Manager): Used to install, manage, and update packages in your project or globally. Packages installed with npm are saved in node_modules and can be imported into your code.
    Example: npm install react
  • npx (Node Package Executor): Executes Node packages without globally installing them. Ideal for running CLI tools or one-off commands, often directly from node_modules/.bin or even downloading and running the package temporarily.
    Example: npx create-react-app my-app runs the CLI without permanently installing it globally.

In short, npm installs packages; npx runs packages.

package.json vs package-lock.json

Both package.json and package-lock.json are part of Node.js project dependency management, but they serve different purposes.

  • package.json: Defines your project metadata and declares the dependencies your project needs (with version ranges). It’s human-readable and used for installing packages, running scripts, and sharing the project.
    Example: Dependencies like "react": "^18.2.0" or devDependencies like "typescript": "^5.1.0".
  • package-lock.json: Records the exact versions of all installed dependencies (including nested sub-dependencies). Ensures consistent installs across machines and environments.
    Automatically generated and updated by npm install, not meant to be edited manually.

In short: package.json declares what you want; package-lock.json locks exactly what is installed.

Purpose of the Dependency Array in useEffect

The dependency array in useEffect tells React when to re-run the effect based on changes to specific values.

  • With dependency array: useEffect(() => { ... }, [dep1, dep2]) The effect runs only when one of the listed dependencies changes. This is useful for fetching data or updating state only when certain values change, preventing unnecessary re-renders or side effects.
  • Empty dependency array: useEffect(() => { ... }, []) Runs once on mount and not on subsequent renders. Ideal for initialization logic like fetching initial data or subscribing to events.
  • No dependency array: useEffect(() => { ... }) Runs on every render. Usually avoided unless the effect must react to every render, as it can hurt performance.

Summary: The dependency array controls the frequency and timing of effect execution, helping optimize performance and avoid unintended behavior.

Purpose of the return Statement in useEffect

The return statement in a useEffect hook defines a cleanup function. This function runs when the component unmounts or before the effect runs again (if dependencies change), allowing you to clean up subscriptions, timers, or event listeners to avoid memory leaks.

  • Example - Event Listener: Add a window resize listener and remove it on unmount. 

useEffect(() => {
  const handleResize = () => {
    console.log(window.innerWidth);
  };

  window.addEventListener('resize', handleResize);

  // Cleanup function
  return () => {
    window.removeEventListener('resize', handleResize);
  };
}, []); // Empty dependency array: runs once on mount

Summary: The return cleanup in useEffect ensures resources are properly released, preventing memory leaks or unintended behavior when components unmount or dependencies change.

Client-Side Routing vs Server-Side Routing

Client-Side Routing is handled in the browser using JavaScript. Instead of requesting a new HTML page from the server for every route change, the app dynamically updates the UI without a full page reload.

  • Used in Single Page Applications (SPAs) like React apps.
  • Navigation feels fast because only data changes, not the entire page.
  • Routing is managed by libraries like React Router.
  • Initial load may be slower because the full app bundle must download first.

Server-Side Routing is handled by the server. Each route change sends a new request to the server, which responds with a completely new HTML page.

  • Traditional multi-page applications (MPAs).
  • Each navigation triggers a full page reload.
  • SEO-friendly by default because HTML is rendered on the server.
  • Can feel slower due to full page refreshes.

Summary: Client-side routing prioritizes speed and smooth UX in SPAs, while server-side routing relies on full page reloads and is naturally SEO-friendly.
