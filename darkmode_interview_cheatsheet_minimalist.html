2. Technical Stack & Core Experience

Describe your preferred stack

  • Frontend: React, Next.js
  • Backend: Node, Express or NestJS (by complexity)
  • SQL and NoSQL; choose by problem

Describe your frontend experience

  • React, Next.js: reusable components, structured state
  • React for interactive UIs; Next.js when SEO, performance, routing matter
  • Accessibility, responsive: semantic HTML, ARIA, Tailwind, Shadcn

Describe your backend experience

  • Node.js: Express (lightweight APIs), NestJS (modular systems)
  • Also Flask → perspective on routing, middleware
  • Focus: REST APIs, validation, auth

Do you prefer frontend or backend development?

  • Full-stack: like seeing frontend and backend connect
  • Comfortable switching layers; enjoy intersection problems
  • Own feature DB → UI, not pick a layer

Have you used [technology] before?

  • Not professionally
  • Learn new stacks quickly; similar patterns → fast ramp-up

When should you use Redux vs local React state?

  • Redux: global state shared across app (e.g. auth)
  • Local state: component-only, avoid unnecessary re-renders
  • Redux: one place for state + update logic, consistent patterns, predictable global state

What are best practices for structuring Redux in large apps?

  • Feature slices in store; simple root reducer
  • Pure reducers; async via Redux Thunk or Redux Toolkit createAsyncThunk
  • No redundant/computed data in store; derive on the fly

How do you design APIs?

  • RESTful: resource-based naming, meaningful status codes
  • JWTs with roles for auth
  • Pagination, filtering, caching from the start; consistent response shapes

How do you structure large-scale React applications?

  • By feature: components, hooks, types per domain
  • Easier to navigate, scale, work in parallel

How do you design APIs for scalability and maintainability?

  • One resource per endpoint (/users, /listings); same fields, consistent types
  • Pagination, filtering for large data
  • Backend: feature modules, shared types, validation, error handling

SQL, ORMs, and database experience

  • Relational DBs: schema design, indexing, transactions
  • Prisma: TypeScript types from schema, type-checked queries, migrations
  • Raw SQL when performance or complexity demands it
  • SQL for integrity-critical; NoSQL for flexible schemas, iteration

UX/UI mindset and examples

  • Intuitive, accessible, easy to navigate
  • Responsive; clear feedback; semantic HTML, ARIA, Tailwind
  • Modular libraries (e.g. Shadcn) for consistency

Server vs client responsibilities

  • Server: data fetch (HTML with content), auth, heavy compute → better UX/security
  • Client: interactivity, client-only state, browser APIs
  • Next.js: Server Components default; client only for hooks, events, dynamic behavior → smaller bundles

State modeling and derived state

  • Minimal source of truth; derive the rest
  • Filter/sort: keep raw list + inputs; display = computed (useMemo or selector) → no sync bugs
  • Don’t duplicate in state what can be computed from props/state

Data fetching and caching strategies

  • One strategy per app: React Query, SWR, or Next.js server-side
  • Loading/error states, deduplication, cache invalidation
  • SSR: fetch at route/layout so data before paint; avoid useEffect fetches everywhere → waterfalls

Hooks design and reusability

  • Extract when reused or component gets hard to follow
  • Clear responsibility, simple API, return only what’s needed; side effects + cleanup inside hook
  • Don’t over-abstract; keep in component until second use case

Performance beyond the component level

  • Layers: bundle (code-split, tree-shake, lazy routes), network (cache, prefetch), render (virtualize lists, memo where profiling shows gain)
  • Measure before/after; don’t memoize by default
  • Server Components + streaming remove client work; React DevTools Profiler for bottlenecks
3. Scalability, Performance & Deployment

Cloud deployment & CI/CD experience

  • Docker for standardized environments; CI/CD to build, test, and deploy reliably
  • GitHub Actions pipeline: backend tests, Docker build, push to ECR, ECS rolling deploy; frontend static build, sync to S3, CloudFront invalidation
  • AWS: ECS (containers), ECR (images), S3 (static), CloudFront (CDN)
  • Explicit rollback: previous ECS task or image for backend; versioned S3 files for frontend
  • After stability, fix the issue and deploy forward through the pipeline

Git experience

  • Feature branches; squash commits; rebase on main
  • PRs for review and knowledge sharing; clear names and descriptions

How do you optimize React for large datasets?

  • Render only what’s visible; avoid huge lists in global state
  • Fetch/stream in chunks; memoize so only changed items re-render

How do you handle API performance under heavy load?

  • Pagination, caching, rate limiting
  • Optimize queries; batching/joins; async for heavy work
  • Logs and metrics to find bottlenecks; indexes/caching before users hit issues
4. Testing, Debugging & Reliability

Testing strategies

  • Jest + React Testing Library: unit tests (components, hooks, UI logic)
  • Integration tests: frontend modules + APIs
  • Postman: real API requests, verify responses
  • Production bugs: fix root cause, add regression tests; focus on critical paths and edge cases

Debugging production issues

  • Logs and monitoring → patterns, error location
  • Trace to layer: frontend, backend, DB
  • Reproduce safely, fix root cause, add tests/validation

How do you monitor production frontend apps?

  • Amazon CloudWatch: logs, errors, page load, API response times
  • Alerts for regressions
  • Focus on critical flows (login, forms)

6. Consultancy & Client Work

Adapting to unfamiliar client codebases

  • Understand domain, structure (routes, data flow, conventions); read tests and recent PRs
  • Small, low-risk changes first; ask for review; document as you go
  • Improve where you work; don’t rewrite for its own sake

Balancing technical quality with delivery timelines

  • Spell out trade-offs: what we defer, what risk
  • Minimum bar: tests on critical paths, error handling, no known security holes
  • Cut scope or simplify before shipping tech debt with no fix path; say when timeline is unrealistic, propose achievable

Communicating trade-offs to non-technical stakeholders

  • Outcomes and consequences, not implementation
  • “If we skip this, risk X, need Y later”; “Option A faster to build, harder to change; B longer now, more flexible”
  • Simple analogies; they understand choice and cost → decide together

Pushing back when requirements are flawed

  • Assume good intent; focus on goal behind requirement
  • Explain downside of ask; suggest alternative that still delivers; offer prototype/spike to validate
  • Align on better outcome; if still disagree, document decision and move on
