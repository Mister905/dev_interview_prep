Deployment Strategy & CI/CD

We used Docker to ensure dev, staging, and production environments were consistent, avoiding “works on my machine” issues. CI/CD pipelines automatically ran tests and built versioned artifacts whenever code was merged to the main branch, catching errors early. The pipelines handled builds, test execution, artifact storage, and controlled promotion from dev to staging to production. Environment variables stored configuration like database URLs, API keys, and feature flags outside the code, keeping each environment's settings separate and secure. Versioned artifacts let us quickly roll back to a stable release if needed, minimizing downtime and protecting service levels.

Debugging

We used structured logging to capture key workflow actions like submission, approval, and disbursement, including claim ID, user ID, and timestamps. Logs were centralized and monitored via dashboards that flagged unusual patterns, such as spikes in failed approvals or slow database responses. When an issue arose, we analyzed the logs to pinpoint the root cause, reproduced it safely, fixed it, and added validations or tests to prevent recurrence, allowing us to proactively resolve problems before users were affected.

Testing

We used a combination of unit tests, integration tests, and CI pipelines to ensure reliability and prevent regressions. Unit tests verified individual functions, like calculating claim eligibility or validating DTO input. Integration tests exercised full workflows end-to-end, such as submission → review → approval → payment, confirming that all business rules were enforced. CI pipelines automatically ran these tests before merges; any failing test blocked the merge, keeping staging and production stable and allowing early detection of issues before they affected users.

Authentication

We implemented robust authentication and access control patterns to protect sensitive workflows. JWT tokens provided stateless authentication, while role-based access control ensured users could only perform actions appropriate to their role (e.g., approvers vs submitters). Middleware and route guards enforced rules consistently on the server side, following the principle of least privilege. Client-side validation was never trusted; all critical checks ran server-side. Access tokens were short-lived for API calls, and refresh tokens allowed securely obtaining new access tokens, minimizing exposure if a token was compromised.

React & Redux: Large-Scale App Structure

  • Organize React apps by feature, grouping components, hooks, and types for better maintainability and scalability
  • useState for local component state and useEffect for side effects (API calls, subscriptions)
  • Use Redux for global state like authentication
  • Split Redux store into feature slices; keep reducers pure (no side effects, no state mutation) and the root reducer simple.
  • Handle async logic using createAsyncThunk from Redux Toolkit.
  • Avoid redundant or computed data in the store; calculate derived values as needed.

API Design, Scalability & Performance

  • RESTful endpoints: GET /users, POST /users, PUT /users/:id
  • Share TypeScript interfaces between backend and frontend: ensures consistent, type-safe API shapes at compile time. Example: interface User { id: number; name: string; email: string }. This prevents type errors in your code but does not stop bad data from reaching the backend.
  • Request validation with DTOs: enforces correct data at runtime on the backend. Example: @IsEmail() email: string; @IsString() name: string. Even if a client bypasses the frontend, DTOs prevent invalid or malicious input from being processed.
  • Secure endpoints (JWT + roles): Only admins can DELETE /users/:id
  • Handle large datasets: GET /posts?page=2&limit=20&authorId=5
  • Optimize queries: SQL join posts with comments; index user_id
  • Feature modules: users/ folder with controller, service, entity
  • Validation & error handling: Return 400 for missing fields, 500 for DB errors
  • API testing with Postman: Systematically test endpoints with valid and invalid payloads, expired JWTs, and edge-case parameters to confirm validation, authorization rules, and error responses behave correctly before release.

React vs Next.js

  • React is a client-side library for building UI, best for highly interactive apps.
  • Next.js is a framework on top of React that adds server-side rendering, static site generation, and routing, which improves SEO and first-load performance.

Why use TypeScript?

  • Adds static typing to catch errors at compile time instead of runtime
  • Makes code easier to read, maintain, and refactor — e.g., if you rename a property in an interface, TypeScript flags all places using it, so you can update safely
  • Ensures consistent types across frontend and backend — e.g., shared User interface used in React and NestJS
  • Improves tooling support with autocomplete and type hints

var vs let vs const - Pass by value vs pass by reference

  • Everything in JavaScript is pass by value, but for objects the value is a reference
    • Pass by value: The function gets a copy of the variable’s value. Changes inside the function do not affect the original variable.
      •           
  function double(x) {
    x = x * 2;
    console.log(x); // 10
  }

  let num = 5;
  double(num);
  console.log(num); // 5 — original not changed
    • Pass by reference: The function gets a reference to the variable. Changes inside the function affect the original variable.
      •           
  function addItem(arr) {
    arr.push("new item");
  }
  
  let myArr = ["item1"];
  addItem(myArr);
  console.log(myArr); // ["item1", "new item"] — original array changed
  • var is function-scoped; let and const are block-scoped
  • Use const by default to prevent reassignment (e.g., API URL, array or object reference)
  • Use let for variables that need to be reassigned (e.g., counter, loop index)
      
function example() {
  if (true) {
    var a = 1; // function-scoped
    let b = 2; // block-scoped
    const c = 3; // block-scoped
  }
      
  console.log(a); // ✅ 1 — 'var' is visible here
  console.log(b); // ❌ Error — 'b' is not defined outside the block
  console.log(c); // ❌ Error — 'c' is not defined outside the block
}

What are microservices and distributed systems?

Microservices break an application into small, independently deployable services, each handling a single business function and communicating via APIs. They were created to address scaling, deployment, and maintainability challenges in monolithic systems. For example, in a JavaScript e-commerce platform, authentication, orders, payments, and inventory could run as separate Node.js services using PostgreSQL and Redis. If order traffic spikes, you can scale just the Orders service with multiple Docker containers instead of the whole application. Other distributed systems exist too. For example, a PostgreSQL cluster with a primary server for writes and read-only replicas to handle high query load, or a Redis caching cluster with data sharded across nodes to speed up access by keeping frequently requested values in memory, are distributed systems but not microservices because they aren't separate business services.

Message Brokers

Message brokers like RabbitMQ allow different parts of a distributed system to communicate asynchronously without blocking each other. This keeps the system responsive even if one service is slow and lets background tasks, such as sending emails or generating reports, run in parallel. I haven't used RabbitMQ professionally yet, but I've built hands-on familiarity independently—for example, in a local Node.js project where I used RabbitMQ to queue email notifications. When a user submitted a form, the app pushed a message to a queue instead of sending the email immediately. Given my experience with similar stacks and architectural patterns, I'm confident I could ramp up quickly and become productive.

SQL vs NoSQL

SQL is a relational database that stores data in structured tables with rows and columns, ideal for complex relationships and strict validation. For example, tracking customer orders, payments, and inventory where relationships and constraints matter. NoSQLis a non-relational database that stores data in documents, key-value pairs, or graphs, which is better for flexible or unstructured schemas. For example, storing user-generated content like social media posts, where some posts may have text, images, videos, or polls, or product reviews where different users include optional ratings, comments, or attachments—so each record can have different fields and structures without breaking the database. 

  // SQL example (relational)
  const usersTable = [
    { id: 1, name: "Alice", email: "alice@example.com" },
    { id: 2, name: "Bob", email: "bob@example.com" }
  ];

  // Query: get user by ID
  const user = usersTable.find(u => u.id === 1);
  console.log(user.name); // Alice

  // NoSQL example (document database)
  const usersCollection = {
    "user_1": { name: "Alice", email: "alice@example.com" },
    "user_2": { name: "Bob", email: "bob@example.com" }
  };

  // Query: get user by key
  console.log(usersCollection["user_1"].name); // Alice

Designing a Chess Game App

  • Key Takeaway: Server is authoritative; clients only send move intents.
  • Core Models / Data: Game, Board, Piece, Player, Move
  • Client Responsibilities: Render board, send move intents (from-square → to-square)
  • Server Responsibilities:
    • Validate moves (legal & correct turn)
    • Update & persist game state
    • Broadcast results to clients via WebSockets
  • Communication / Sync: WebSockets push updated state to all clients in real-time
  • Security / Rules Enforcement: Auth identifies player; server enforces all game rules
  • Scaling / Performance Notes: Minimal, mostly focused on real-time sync

Designing Twitter

  • Key Takeaway: Server is authoritative; clients read & render only.
  • Core Models / Data: User, Post, Follow, Feed
  • Client Responsibilities: Render feed & profiles, send post requests
  • Server Responsibilities:

  • Scaling / Performance Notes:

Designing a Real-Time Auction Platform

  • Key Takeaway: Backend is authoritative; clients submit bids and receive updates in real-time.
  • Core Models / Data: User, Auction, Item, Bid, BidHistory, AuctionStatus
  • Client Responsibilities: Display current auctions, submit bid intents, render real-time bid updates
  • Server Responsibilities:
    • Validate incoming bids (amount, timing, user eligibility)
    • Update and persist auction state, including highest bid and bidder
    • Broadcast auction updates to subscribed clients via WebSockets or server-sent events
    • Handle concurrent bids safely to prevent race conditions
    • Support creation and management of auctions, items, and bidding rules
  • Communication / Sync: WebSockets for live bid updates; REST APIs for auction creation, user management, and historical queries
  • Security / Rules Enforcement: Auth identifies bidders; server enforces bid validity, auction timing, and anti-cheat measures
  • Scaling / Performance Notes:
    • Use in-memory caching (Redis) for current auction state to reduce DB hits
    • Queue concurrent bid events to ensure sequential processing
    • Horizontally scale WebSocket servers with pub/sub for broadcasting updates
