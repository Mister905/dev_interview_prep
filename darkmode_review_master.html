How do you balance speed vs code quality?

How do you decide whether logic belongs on the frontend or backend?

I put logic on the backend if it involves security, data validation, or business rules that shouldn't be exposed. Frontend logic is for UI behavior, client-side validation for UX, and formatting. I consider performance: complex calculations that don't need real-time updates can run on the backend. I also think about maintainability: if multiple clients need the same logic, it belongs on the backend. The rule of thumb is: if it can be manipulated or bypassed on the frontend, it needs backend validation.

What tradeoffs do you consider when choosing a tech stack?

I consider team expertise, community support, performance requirements, and long-term maintainability. A newer framework might offer better DX but have less documentation. A mature stack has more resources but might be less flexible. I also think about hiring: popular stacks are easier to hire for. For startups, I prioritize speed and flexibility. For established products, stability and maintainability matter more. The key is matching the stack to the project's needs and constraints.

How do you approach learning a new framework or language quickly?

I start with official documentation and build a small project to understand core concepts. I focus on the framework's unique patterns rather than trying to apply old patterns. I read example code and common patterns, then practice by building something similar to what I'll work on. I use the framework's recommended tooling and conventions rather than fighting them. The goal is to understand the "why" behind the framework's design, not just the syntax.

What is semantic HTML?

Elements like <article>, <section>, <nav> improve accessibility, SEO, and clarity.

How do you use <label> for accessibility?

Links labels to controls, improves screen reader support and click targets.

How does CSS specificity and conflict resolution work?

Specificity order: inline > ID > class > element. Avoid !important, use DevTools to debug.

What is the CSS box model?

Content, padding, border, margin. Padding affects inside space, margin outside space.

How do you create responsive layouts?

Media queries, fluid units, Flexbox/Grid.

What are the differences between em, rem, and px?

px fixed, em relative to parent, rem relative to root.

How does the browser render a webpage from HTML/CSS/JS?

The browser parses HTML to build the DOM tree, then parses CSS to build the CSSOM. It combines them into a render tree, calculates layout (reflow), and paints pixels to the screen. JavaScript can modify the DOM, triggering reflow and repaint. The browser optimizes by batching changes and using layers for animations. Critical rendering path optimization involves minimizing render-blocking resources, deferring non-critical CSS/JS, and using efficient selectors.

What performance issues have you encountered on the frontend, and how did you fix them?

I've dealt with slow initial load times from large bundle sizes. I fixed it by code-splitting routes, lazy loading components, and optimizing images. Another issue was unnecessary re-renders causing UI lag. I used React.memo, useMemo, and useCallback to prevent re-renders, and moved state down to components that actually needed it. I also optimized API calls by debouncing search inputs and caching responses. Performance monitoring tools helped identify the bottlenecks.

Explain state management in a frontend app. When is global state necessary?

State management handles data flow in the app. Local state (useState) works for component-specific data. Global state (Redux, Context) is needed when multiple components share data or when state needs to persist across navigation. I use global state for user authentication, theme preferences, and data that multiple pages need. For everything else, I prefer local state or prop drilling to avoid unnecessary complexity. The key is not over-engineering: start local, lift state up when needed, and only go global when it's clearly necessary.

How do you handle forms, validation, and user feedback?

I use controlled components with React state and validate on both client and server. Client-side validation provides immediate feedback for UX, while server-side ensures security. I show validation errors inline, near the relevant field, and disable submit until the form is valid. For complex forms, I use libraries like Formik or React Hook Form. I provide clear error messages and highlight invalid fields. Loading states during submission prevent double-submits, and success messages confirm completion.

What causes unnecessary re-renders, and how do you prevent them?

Re-renders happen when props or state change. Common causes: creating new objects/arrays in render, passing inline functions as props, or state updates triggering parent re-renders. I prevent them by memoizing values with useMemo, callbacks with useCallback, and components with React.memo. I also move state down to the lowest component that needs it, and use refs for values that don't affect rendering. The React DevTools Profiler helps identify what's causing re-renders.

How do you make an app accessible (a11y)?

I use semantic HTML, proper ARIA labels, and keyboard navigation. All interactive elements are keyboard-accessible, and I maintain focus management. I ensure sufficient color contrast, provide text alternatives for images, and use proper heading hierarchy. Forms have associated labels, and errors are announced to screen readers. I test with keyboard-only navigation and screen readers. Following WCAG guidelines ensures the app is usable by people with disabilities.

UX / UI Engineering

How do UX decisions impact backend or database design?

UX requirements often drive backend architecture. For example, real-time features require WebSockets or polling. Search functionality needs database indexing strategies. Pagination affects API design and query optimization. User permissions determine authorization logic. Loading states might require backend endpoints that return partial data. The key is collaboration: understanding UX goals helps design efficient backend solutions, and backend constraints inform what's feasible for UX.

How do you ensure consistency in UI components?

I use a design system or component library with documented patterns. I create reusable components with consistent props and styling. I establish spacing, typography, and color tokens that all components use. Code reviews catch inconsistencies, and I refactor similar components to share common logic. Having a style guide or Storybook helps maintain consistency across the team. The goal is that similar features look and behave the same way throughout the app.

How do you balance usability vs performance?

I prioritize performance for core user flows, but don't sacrifice usability for minor optimizations. For example, I'll optimize critical rendering path but keep helpful animations. I use progressive enhancement: ensure core functionality works fast, then add enhancements. Loading states and skeleton screens maintain perceived performance while data loads. I measure real performance with tools like Lighthouse and get user feedback on perceived speed. The balance depends on the feature's importance and user expectations.

How do you design forms to minimize user errors?

I use clear labels, helpful placeholders, and appropriate input types. I validate in real-time and show errors immediately, not just on submit. I group related fields, use progressive disclosure for complex forms, and provide examples for ambiguous inputs. I disable submit until required fields are valid and show character counts for limited inputs. Clear error messages explain what's wrong and how to fix it. The goal is preventing errors before they happen, not just catching them.

How do you handle loading states, errors, and empty states?

Loading states use skeleton screens or spinners to show progress. Errors display clear messages with actionable next steps, and I provide retry options when appropriate. Empty states explain why there's no content and suggest actions, like "No results found. Try adjusting your filters." I handle different error types: network errors, validation errors, and server errors each get appropriate messaging. These states are part of the design, not afterthoughts, and they guide users through the experience.

What accessibility standards do you follow (WCAG basics)?

I follow WCAG 2.1 Level AA guidelines. This includes: sufficient color contrast (4.5:1 for text), keyboard navigation for all interactive elements, screen reader compatibility with semantic HTML and ARIA, text alternatives for images, and proper heading hierarchy. I ensure forms are accessible with labels and error announcements. I test with keyboard navigation and screen readers. The goal is making the app usable by everyone, regardless of ability.

Backend & Infrastructure

How do you design a Node.js API?

I start by clarifying the use cases and data involved, then design RESTful endpoints around clear resources. Controllers are kept thin, business logic goes into services, and a data access layer or ORM handles database interactions. Consistent request/response shapes, proper HTTP status codes, and centralized error handling make the API predictable and maintainable.

How do you design a REST API? What makes one "good"?

A good REST API uses clear resource-based URLs, proper HTTP verbs (GET, POST, PUT, DELETE), and consistent response formats. It follows REST principles: stateless requests, cacheable responses, and layered architecture. Good APIs have clear documentation, versioning strategy, and predictable error handling. They're designed for the client's needs, not just mirroring the database structure. Proper status codes, pagination for lists, and filtering/search capabilities make APIs more usable.

How do you handle errors and ensure reliability in Node?

Error handling is part of the API contract: validate inputs early, use structured error responses, centralize handling. Log enough context to diagnose production issues. Services fail gracefully rather than crashing.

Principle: use try/catch with async/await for async operations.

How do you think about scalability and performance?
  • Build clear and correct functionality first.
  • Optimize using efficient queries, proper indexing, minimal work in handlers.
  • Design stateless services for horizontal scaling.
  • Use caching or batching only when real performance needs exist.

How do you structure a Node.js project?
  • Routes handle HTTP wiring.
  • Controllers handle request/response logic.
  • Services contain business rules.
  • Data layer/ORM handles persistence.
  • Shared utilities and configuration are centralized.

How do you structure a Node.js backend for scalability?

I design stateless services that can run multiple instances behind a load balancer. I separate concerns: routes, controllers, services, and data layers are independent. I use connection pooling for databases and implement caching strategies. I avoid storing session state in memory and use external stores like Redis. I design for horizontal scaling by keeping services independent and using message queues for async processing. Monitoring and logging help identify bottlenecks as the system grows.

What's the difference between blocking and non-blocking code in Node?

Blocking code stops execution until an operation completes, preventing other requests from being handled. Non-blocking code uses callbacks, promises, or async/await to continue processing other requests while waiting for I/O operations. Node's event loop handles this: it processes non-blocking operations asynchronously, allowing the single thread to handle many concurrent requests. Blocking operations like synchronous file reads or CPU-intensive tasks can hurt performance, so I use async alternatives and offload heavy computation when needed.

When would you choose REST vs GraphQL?

I choose REST for simple CRUD operations, caching needs, and when the API structure is straightforward. REST is easier to understand, has better tooling, and works well with HTTP caching. I choose GraphQL when clients need flexible data fetching, when over-fetching is a problem, or when multiple clients need different data shapes. GraphQL is better for complex relationships and reducing round trips, but adds complexity. The decision depends on the use case: REST for simplicity, GraphQL for flexibility.

How do you validate and sanitize user input?

I validate input structure and types using libraries like Joi or Zod, checking required fields, formats, and ranges. I sanitize by escaping special characters, removing potentially harmful content, and using parameterized queries to prevent SQL injection. I validate on both client and server, but never trust client-side validation alone. I whitelist allowed values rather than blacklisting, and use type coercion carefully. The goal is ensuring data is safe and in the expected format before processing.

Explain middleware and when you'd use it.

Middleware are functions that run between the request and response, modifying the request/response or performing side effects. I use middleware for authentication, logging, error handling, request parsing, and CORS. They execute in order and can pass control to the next middleware or end the request. Common use cases: validating tokens, parsing JSON bodies, logging requests, handling errors centrally. Middleware keeps handlers focused on business logic by extracting cross-cutting concerns.

How do you structure a backend project for maintainability?

I organize by feature or layer, with clear separation of concerns. Routes define endpoints, controllers handle HTTP logic, services contain business rules, and repositories handle data access. I keep functions small and focused, use consistent naming, and document complex logic. I extract shared utilities and configuration. Testing is built in, not added later. Code reviews and linting maintain quality. The structure should make it easy for new developers to find and understand code.

What are common backend performance bottlenecks?

Common bottlenecks include slow database queries without proper indexing, N+1 query problems, inefficient algorithms, blocking I/O operations, and memory leaks. Other issues: lack of caching, over-fetching data, synchronous operations in async contexts, and not using connection pooling. I identify bottlenecks through profiling, monitoring, and logging. Solutions include optimizing queries, adding indexes, implementing caching, using async operations, and scaling horizontally when needed.

How do you handle authentication?

Token-based authentication with JWTs, validated via middleware on protected routes. Secrets and token lifetimes configurable. Authorization checks enforced at the service level.

  • Refresh tokens
  • Role-based access control
  • Secure storage of secrets

How do you test APIs?

Focus on API behavior rather than implementation. Integration tests hit endpoints with known inputs and assert responses, status codes, and side effects. Unit tests added for complex logic at the service level.

Tool: Jest

How do you deploy Node services?

CI/CD pipeline builds, tests, and deploys to cloud (AWS/Azure). Environment variables for configuration, stateless services for safe redeployment, monitoring of deployments, rollbacks when necessary.

How do you design a database schema?

Understand entities and relationships. Normalize schema. Choose types carefully, enforce integrity with constraints, and keep it flexible to evolve. Revisit design based on real usage.

How do you optimize database performance?
  • Optimize only when necessary (slow queries, over-fetching).
  • Use indexes and efficient query structures.
  • Pagination and query planning considered.

How do you use ORMs effectively?

Use ORM for maintainable, consistent data access. Drop to raw queries for performance when needed.

How do you handle migrations and schema changes?

Versioned migrations, reviewed carefully, applied via deployment workflows. Risky changes planned to minimize downtime.

When would you choose SQL vs NoSQL?

SQL when relationships and consistency matter. NoSQL for flexibility or document-based data. Decision driven by access patterns, not preference.

How do you decide between SQL and NoSQL?

I choose SQL when I need ACID transactions, complex relationships, or structured data with clear schema. SQL is better for financial data, user accounts, or anything requiring consistency. I choose NoSQL for flexible schemas, horizontal scaling needs, or document-based data like user profiles or content. NoSQL works well for high-volume writes or when schema changes frequently. The decision is based on data access patterns, consistency requirements, and scaling needs, not just preference.

Explain indexing and how it improves performance.

Indexes are data structures that allow the database to find rows quickly without scanning the entire table. They're like a book's index: instead of reading every page, you jump to the right section. Indexes speed up WHERE clauses, JOINs, and ORDER BY operations. However, they slow down writes because the index must be updated. I create indexes on frequently queried columns, especially foreign keys and search fields. I monitor query performance and add indexes when queries are slow, but avoid over-indexing which hurts write performance.

What is normalization? When might you not normalize?

Normalization reduces data redundancy by organizing data into related tables. It prevents update anomalies and saves storage. I normalize to third normal form for most cases. However, I might denormalize for performance: when joins are expensive, when read performance is critical, or when data is mostly read-only. Denormalization trades storage and potential inconsistency for speed. The decision depends on read/write patterns: normalize for consistency, denormalize for performance when needed.

How do you handle migrations?

I use versioned migration files that are applied in order. Migrations are reviewed like code changes and tested in staging first. For risky changes like dropping columns, I create them in steps: add new column, migrate data, update code, then remove old column. I always backup before migrations and have a rollback plan. Migrations run as part of the deployment process, and I monitor for issues. The goal is making schema changes safe, reversible, and trackable.

Describe a slow query you've optimized.

I once had a query joining multiple tables that took several seconds. I used EXPLAIN to see the execution plan and found it was doing full table scans. I added indexes on the join columns and WHERE clause filters. I also restructured the query to avoid a subquery that was executing for each row. The query went from 5 seconds to under 100ms. I learned to always check the execution plan and add indexes strategically, not just on every column.

How do transactions work, and when are they important?

Transactions group multiple database operations into a single atomic unit: either all succeed or all fail. They ensure ACID properties: Atomicity, Consistency, Isolation, Durability. Transactions are critical when operations must happen together, like transferring money between accounts or updating related records. I use transactions for financial operations, maintaining referential integrity, and any multi-step operation that must be all-or-nothing. Without transactions, partial failures can leave data in an inconsistent state.

Testing & Quality

What types of tests do you write (unit, integration, e2e)?

I write unit tests for individual functions and components, testing logic in isolation. Integration tests verify that components work together, like API endpoints with databases. E2E tests cover critical user flows from start to finish. I use more unit tests (fast, easy to maintain) and fewer E2E tests (slower, more brittle). The testing pyramid: many unit tests, some integration tests, few E2E tests. Each type catches different issues and has different maintenance costs.

How do you decide what's worth testing?

I prioritize testing business logic, edge cases, and critical paths. I test functions with complex logic, user-facing features, and code that's likely to break. I don't test implementation details, third-party libraries, or trivial code. The goal is confidence that the app works correctly, not 100% coverage. I focus on tests that catch real bugs and provide value when refactoring. High-value tests are those that would catch issues users would notice.

System Design

How would you design a simple chat application?

I'd use WebSockets for real-time messaging between clients and server. The backend would handle message routing, user presence, and message storage. I'd store messages in a database with indexes on user IDs and timestamps. For scaling, I'd use a message queue to handle high volumes and multiple server instances. The frontend would connect via WebSocket, display messages in real-time, and handle reconnection logic. I'd add features like message history, typing indicators, and read receipts incrementally. The key is real-time communication and reliable message delivery.

How would you handle file uploads at scale?

I'd use cloud storage like S3 or Azure Blob Storage instead of storing files on the server. The backend would generate signed URLs for direct client-to-storage uploads, reducing server load. I'd implement chunked uploads for large files and validate file types and sizes. I'd use a CDN to serve files efficiently. For processing, I'd use background jobs to handle image resizing or video encoding. The database would store file metadata and references, not the files themselves. This approach scales horizontally and handles large files efficiently.

How would you design an API for a to-do app used by thousands of users?

I'd design RESTful endpoints: GET/POST /todos, GET/PUT/DELETE /todos/:id. I'd use pagination for listing todos and filtering for completed/pending. Authentication would use JWTs, and each user would only access their own todos. I'd use database indexes on user_id and created_at for performance. I'd implement rate limiting to prevent abuse. The API would return consistent JSON responses with proper status codes. For scaling, I'd use caching for frequently accessed data and consider read replicas for the database. The design prioritizes security, performance, and clear structure.

How would you prevent duplicate requests or race conditions?

I'd use idempotency keys: clients send a unique key with requests, and the server processes each key only once. I'd disable buttons during submission to prevent double-clicks. On the backend, I'd use database constraints or transactions to prevent duplicate operations. For race conditions, I'd use optimistic locking with version numbers or pessimistic locking for critical sections. I'd also implement request deduplication by caching recent request signatures. The goal is ensuring operations happen exactly once, even if requests are duplicated.
