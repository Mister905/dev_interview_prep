Next.js Fundamentals for React Developers

1. Project Structure

  • pages/ – React components used as routes automatically
  • app/ (Next 13+) – React Server Components and nested layouts
  • components/ – Reusable React components (same as in React projects)
  • public/ – Static assets like React’s public/
  • styles/ – CSS Modules, Tailwind, or global CSS just like in React
  • next.config.js – Extra configuration for routing, images, environment variables

2. Routing

  • File-based routing: pages/index.js → / (React does not handle routing automatically; Next.js builds on React Router concepts)
  • Dynamic routes: pages/posts/[id].js → /posts/123 (React Router would use :id params)
  • Catch-all routes: pages/blog/[...slug].js
  • Linking: <Link href="/about">About</Link> (similar to React Router’s <Link to="/about">)
  • App Router (Next 13+): nested layouts (like React layouts, but automatic and SSR-aware)

3. Data Fetching

  • getStaticProps() – pre-render data at build time (like fetching before render in React apps using props)
  • getServerSideProps() – fetch data on every request (similar to React useEffect + fetch but server-side)
  • getStaticPaths() – define dynamic paths for SSG pages (like mapping data to routes in React Router)
  • App Router async components: fetch directly inside React server components instead of useEffect
  • Client-side fetching: SWR / React Query (same as client-side React fetch patterns)
  • React dev note: Server vs Client Components is unique; client components behave like normal React components

4. API Routes

  • pages/api/hello.js → /api/hello (like mini Express endpoints inside your React project)
  • Supports GET, POST, etc. via req.method
  • Server-only code, safe for env variables
  • React devs: API routes let you stay in Next.js ecosystem without a separate backend 
    • export default function handler(req, res) {
  if (req.method === 'POST') {
    res.status(200).json({ message: 'Hello from API' })
  }
}

5. Styling

  • Global CSS, CSS Modules, Tailwind — identical usage as in React projects
  • Styled JSX / styled-components — React devs already familiar with these
  • Next.js just ensures efficient bundling

6. Next.js Features for React Developers

  • <Image /> component — optimized images (like using <img /> in React but with automatic optimization))
  • Head component for meta tags (like react-helmet))
  • Automatic code splitting & prefetching (React devs manually split bundles with React.lazy / Suspense)
  • Server Components (React components, but run on server for lighter client bundles))
  • Middleware / edge functions (similar to server-side logic in Express + React, but integrated))
  • Fast Refresh (like React HMR in CRA or Vite)

7. Deployment

  • Vercel first-class (like hosting a CRA app on Netlify, but SSR-ready)
  • Static export: next export (like React static export build)
  • Environment variables: .env.local, .env.production (same as CRA env files)
